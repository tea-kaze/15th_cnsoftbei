## 一、需求场景分析

### 1.1 行业痛点

| 痛点 | 描述 |
|------|------|
| 导游资源稀缺 | 黄金周/旺季专业导游供不应求，游客体验大打折扣 |
| 信息单向传递 | 传统录音导览设备内容固定，无法与游客互动，解答个性化问题 |
| 缺乏情感连接 | 冰冷设备难以提供如真人导游般的亲切感和情感互动 |
| 管理盲区 | 景区管理者难以量化评估导游服务质量，无法精准获取游客真实反馈 |

### 1.2 解决方案

本系统构建一个 **7×24小时在线AI数字人导游**，游客通过手机App或景区终端与数字人交互，获得个性化导览服务；同时为管理方提供数据看板，助力科学决策。

### 1.3 用户角色

| 角色 | 需求 |
|------|------|
| 游客 | 语音/文字提问 → 获得准确的景区知识问答 → 收到个性化路线推荐 |
| 景区管理员 | 管理知识库文档 → 查看服务数据大屏 → 获取游客反馈报告 → 配置数字人形象 |

### 2.2 技术栈

| 层面 | 技术选型 | 理由 |
|------|---------|------|
| 前端框架 | Vue 3 + Vite + Element Plus | 轻量、中文生态好、移动端响应式 |
| 图表 | ECharts 5 | 国产、文档完善、管理后台标配 |
| 语音识别 | Web Speech API (浏览器端) | 零成本、Chrome/Edge 内置、中文识别率高 |
| 语音合成 | edge-tts (Python) | 免费、中文质量极高、40+发音人 |
| LLM | DeepSeek-chat | 性价比高、中文能力强、支持流式生成 |
| 向量检索 | sklearn TF-IDF (char ngram 1-3) | 离线可用、无需 GPU、竞赛环境友好 |
| 数字人 | CSS 2D + VTube Studio (可选) | CSS 保底可用，VTS 3D 提升表现力 |
| 分析数据库 | SQLite | Python 内置、零配置、竞赛场景足够 |

### 2.3 数据流（游客问答）

```
1. 游客语音 → Web Speech API → 文字
2. 文字 → POST /api/chat/ask/stream → TF-IDF 检索 top-5 → DeepSeek LLM 流式生成
3. 每个 token 通过 SSE 实时推送到前端 → 打字机效果显示
4. 流式完成后 → edge-tts 合成语音 → 前端播放
5. 同时检测情绪关键词 → 触发 CSS 数字人表情 / VTube Studio 表情
6. 交互日志 → SQLite analytics.db
```

## 三、模块设计

### 3.1 后端模块

#### 3.1.1 RAG 检索引擎 (`services/rag.py`)

**算法**: TF-IDF + 余弦相似度

```python
# 向量化配置
TfidfVectorizer(analyzer="char", ngram_range=(1, 3), max_features=2000)

# 检索流程:
# 1. 查询文本 → vectorizer.transform() → query_vector
# 2. cosine_similarity(query_vector, doc_matrix) → 得分数组
# 3. 取 top_k 个得分 > 0 的结果
```

**文档处理**:
- `.txt`: 直接 UTF-8 读取
- `.pdf`: pypdf 逐页提取文本
- `.docx`: python-docx 提取段落 + 表格（表格用 `|` 分隔）

**分块策略**: 按段落合并，目标 500 字符/块，保持段落边界完整

**持久化**: 全量存储为 `knowledge_index.json`，启动时自动加载重建索引

#### 3.1.2 LLM 服务 (`services/llm.py`)

**模型**: DeepSeek-chat（通过 OpenAI SDK 兼容接口）

**系统提示词设计**:
```
角色：景区AI数字人导游
要求：
1. 亲切热情的语气，像真人导游
2. 只根据知识库内容回答，不编造
3. 知识不足时礼貌告知并给出建议
4. 简洁明了，适合语音播放
5. 结合对话历史理解代词指代
6. 用**重点**加粗关键信息，空行分段
```

**流式生成**:
```python
def answer_stream(question, context_chunks, history=None):
    stream = client.chat.completions.create(
        model=MODEL, messages=messages,
        temperature=0.7, max_tokens=500, stream=True
    )
    for chunk in stream:
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content
```

#### 3.1.3 TTS 语音合成 (`services/tts_service.py`)

**引擎**: edge-tts（微软 Edge 浏览器 TTS 引擎）

**可用语音**:
| 语音 | 性别 | 风格 |
|------|------|------|
| 晓晓 (Xiaoxiao) | 女 | 温暖 |
| 云希 (Yunxi) | 男 | 新闻 |
| 晓伊 (Xiaoyi) | 女 | 活泼 |
| 云健 (Yunjian) | 男 | 运动 |
| 云扬 (Yunyang) | 男 | 新闻 |

**缓存**: 以 `MD5(text+voice+rate+pitch)` 为键缓存 MP3 文件，命中时直接返回

**字词时间戳**: 按标点分词，按字符数比例均匀分配时间（近似方法，满足口型同步需求）

#### 3.1.4 VTube Studio 集成 (`services/vts_service.py`)

**通信**: WebSocket 连接 `ws://127.0.0.1:8001`

**能力映射**:
| API | 功能 | 应用 |
|-----|------|------|
| ExpressionActivationRequest | 激活内置表情 | 情绪关键词→表情联动 |
| InjectParameterDataRequest | 设置 Live2D 参数 | MouthOpen 口型同步 |
| HotkeyTriggerRequest | 触发热键动画 | 挥手欢迎/鞠躬感谢 |

**口型同步线程**: 每 50ms 根据字词时间戳计算 MouthOpen 值（0.0~1.0），通过 WebSocket 推送到 VTS

**chat_logs 表**:

#### 3.1.6 推荐引擎 (`services/recommend_service.py`)

**算法**: 关键词分类 + 路线匹配

**三条预设路线**:
| 路线 | 时长 | 景点 | 关键词触发 |
|------|------|------|-----------|
| 历史文化深度路线 | 6h | 12站 | 历史/佛教/古迹/文化 |
| 自然风光爱好者路线 | 5h | 8站 | 自然/风景/摄影/太湖 |
| 亲子家庭欢乐路线 | 4h | 7站 | 亲子/孩子/家庭/轻松 |

**分类逻辑**: 对用户输入进行关键词评分，最高分类别即为推荐路线；无匹配时默认推荐历史文化路线。

**路线配置**: 路线数据已从代码中抽离至 `routes_config.json`，景区运营人员可直接编辑 JSON 文件自定义景点、时长和提示语，无需修改 Python 代码。

#### 3.1.7 大数据分析 (`services/analytics_service.py`)

基于赛方提供的 14 万条景区旅游行为数据（含 522 条灵山胜境记录），为管理后台提供游客画像与消费分析：

- **年龄分布**: 18-25 / 26-35 / 36-45 / 46-60 / 60+
- **性别比例**、**团队规模**（独行/小团/中团/大团）
- **消费分析**: 门票/餐饮/购物/交通/娱乐人均支出
- **月度趋势**: 各月访问量分布，识别高峰月份
- **停留时长**: 平均景区停留时间
- **满意度分布**: 基于原始数据的满意度标签统计

数据通过 `openpyxl` 读取 xlsx 文件，首次加载后缓存至 `analytics_cache.json`。

#### 3.1.8 安全中间件 (`middleware.py`)

**管理后台鉴权**: POST/PUT/DELETE/PATCH 操作需携带 `X-Admin-Key` 请求头，Key 通过 `.env` 文件中 `ADMIN_API_KEY` 配置。

## 五、核心算法

### 5.1 TF-IDF 检索

```
Input:  query (用户问题)
Output: top-k 相关文档块

1. query_vector = vectorizer.transform([query])
2. scores = cosine_similarity(query_vector, doc_matrix)[0]
3. ranked = sorted(enumerate(scores), key=lambda x: x[1], reverse=True)
4. return [documents[i] for i, score in ranked[:k] if score > 0]
```

**为什么不用 ChromaDB/FAISS？**
中国大陆网络无法访问 HuggingFace 下载 embedding 模型；sklearn TF-IDF 纯离线运行，且中文 char-ngram 对短文本检索效果好。

### 5.2 情绪检测

```
输入: LLM 回答文本
输出: surprised | happy | sad | neutral

关键词匹配:
  surprised ← 惊人/震撼/高达/最大/第一/奇迹/88米
  happy     ← 欢迎/开心/推荐/美好/精彩/壮丽/美丽
  sad       ← 抱歉/遗憾/可惜/难过
  angry     ← （待后端补充触发词）
  无匹配    → neutral
```

### 5.3 路线推荐分类

```
输入: 用户兴趣描述
输出: history | nature | family

对每个类别计算关键词命中次数，取最高分；
无匹配时默认返回 history（历史文化路线）。
```

### 5.4 Markdown 渲染（前端）

```
输入: LLM 回答（含 **bold** 和空行）
输出: HTML

1. **text** → <strong>text</strong>
2. \n\n    → </p><p>  (paragraph break)
3. \n      → <br>       (line break)
4. wrap in <p>...</p>
```
